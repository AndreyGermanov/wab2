Создание резервных копий и восстановление системы при сбоях

Введение

Сервер содержит средства для создания резервных копий системы и данных и для восстановления системы в случае сбоев. Рекомендуется периодически делать резервные копии системы, а создание резервных копий данных поставить на регулярную основу.

Резервные копии системы и резервные копии данных создаются по разному.

Создание резервной копии системы и восстановление системы из резервной копии
В комплекте с сервером поставляется CD-диск аварийного восстановления системы. С помощью CD-диска аварийного восстановления можно загрузить сервер после сбоя, после которого система перестала загружаться. С помощью этого диска можно выполнить действия, которые невозможно выполнить в обычном режиме работы сервера.

Диск аварийного восстановления позволяет выполнять следующие действия:
  • Создание резервных копий системы сервера
  • Восстановление системы из резервной копии
  • Изменение размера разделов жесткого диска
  • Инсталляция нового жесткого диска в RAID-массив
  • Переустановка менеджера загрузки
В этой теме мы рассмотрим все это, кроме "Инсталляция нового жесткого диска в RAID-массив" и "Переустановка менеджера загрузки". Инсталляцию нового жесткого диска в RAID-массив должен осуществлять сервисный инженер ООО "ЛВА". Он же после этого переустанавливает менеджер загрузки.


Для работы с утилитами CD-диска аварийного восстановления нужно загрузить сервер с этого диска и войти в него (имя пользователя root, пароль 111111).

На рабочем столе содержатся следующие полезные утилиты:
  • Восстановление системы - утилита для создания резервных копий системы, восстановления системы из резервной копии, инсталляции нового жесткого диска в RAID-массив и переустановки менеджера загрузки
  • Данные разделов - предоставляет доступ к данным разделов, которые находятся в двух папках: SYSTEM - данные системного раздела, DATA - данные раздела данных. Это позволяет, например, сделать резервную копию данных из раздела данных или вручную на системном разделе найти и исправить конфигурационные файлы, неправильная работа с которыми привела систему в неработоспособное состояние
  • Управление логическими томами - позволяет изменять размеры разделов жесткого диска
  • Webmin - позволяет воспользоваться интерфейсом Webmin прямо с диска аварийного восстановления для того, чтобы посмотреть состояние RAID-массива и восстановить его.
Все операции связаны с жестким диском и прежде чем их выполнять, нужно понять, что представляет из себя жесткий диск сервера.

Структура жесткого диска сервера



Схематично диск сервера состоит из главной загрузочной записи (1) и всей остальной области диска (2,3,4).

Главная загрузочная запись это первые 512 байт жесткого диска. В главной загрузочной записи находится менеджер загрузки GRUB (1.1) и таблица разметки диска на разделы (1.2). Сервер использует нестандартный механизм разметки диска. Вместо того чтобы напрямую с помощью fdisk разбить жесткий диск на разделы для системы, данных и области подкачки, мы создали всего один раздел, равный объему всего диска и передали его под управление Менеджера Логических Томов LVM. Менеджер логических томов LVM это расширение системы, предоставляющее множество дополнительных возможностей по работе с разделами, по сравнению со стандартными возможностями fdisk. Раздел, который мы создали, был передан под управление LVM как "физический том LVM".

Физический том LVM был включен в "группу томов LVM", которая называется vg0, а в ней были созданы три логических тома LVM: SYSTEM, DATA и SWAP. Операционная система видит эти логические тома как обычные разделы жесткого диска, которые представлены дисковыми устройствами /dev/vg0/SYSTEM, /dev/vg0/DATA и /dev/vg0/SWAP. С помощью стандартных средств системы, раздел SYSTEM примонтирован как корневая файловая система, а раздел DATA примонтирован к каталогу /data. Раздел SWAP используется в качестве раздела подкачки (виртуальной памяти). Одно из основных преимуществ LVM заключается в том, что эти разделы являются динамическими. Их размер не фиксирован, что позволяет очень быстро уменьшать и увеличивать разделы. Для этой цели используется утилита "Управление логическими томами", которая находится на рабочем столе на CD-диске аварийного восстановления.

ПРИМЕЧАНИЕ: В некоторых конфигурациях раздела подкачки (SWAP) может не быть. Вместо него используется файл подкачки, который называется swap. Он находится в корне раздела SYSTEM.

Работа с утилитой "Восстановление системы"

Создание резервных копий системы

Как видно из схемы, система и данные отделены друг от друга, что позволяет отдельно создавать резервные копии системы и отдельно создавать резервные копии данных и затем отдельно их восстанавливать. Для создания резервных копий системы используется функция утилиты "Восстановление системы", которая находится на CD-диске аварийного восстановления. Эта функция включает в резервную копию все, кроме разделов DATA и SWAP.  В резервную копию входят все компоненты, определяющие структуру диска: главная загрузочная запись (области 1, 1.1, 1.2) и таблица разделов менеджера логических томов (облать 2), а также образ раздела SYSTEM (область 3).

Резервные копии системы нужно делать для двух целей:

1. Чтобы восстановить систему после серьезного сбоя, который привел к потере данных на системном разделе или до него (в MBR, например)
2. Чтобы вернуть систему в рабочее состояние после неумелой настройки, которая привела систему в худшее состояние и "чинить" ее вручную не представляется возможным или слишком трудоемко.

Создавать резервную копию системы нужно каждый раз после внесения серьезных изменений в нее, но только после проверки работатоспособности системы после этих изменений. Также, обязательно нужно создавать резервную копию системы после изменения размеров разделов, так как это вносит изменения в таблицу разделов менеджера логических томов (2).

Предполагается, что резервная копия системы должна записываться на внешний носитель (например на внешний диск USB), поэтому внешний носитель нужно подключить до запуска процедуры создания резервной копии.
Процедура создания резервной копии системы.
1. Запустите утилиту "Восстановление системы"
2. Выполните команду "Создать резервную копию системы" из меню


3. Найдите устройство, на которое нужно записать резервную копию, найдите на нем папку, в которой будет создана резервная копия, выберите ее (не входя в эту папку) и нажмите ОК.


4. Начнется процедура создания резервной копии системы. Для увеличения производиельности, мы не включали никакого индикатора, визуализирующего этот процесс. Процесс будет идти незаметно. Дождитесь завершения процедуры создания резервной копии системы, о чем сигнализирует сообщение "Создание резервной копии успешно завершено". Обычно процедура длится не более 10 минут. Резервная копия системы в базовом варианте занимает не более 3 Гб.

При создании резервной копии, утилита создает в указанном каталоге следующие файлы:
  • sd*.mbr - образы главных загрузочных записей всех жестких дисков
  • md0_uuid - уникальный идентификатор RAID-массива
  • md0_name - имя RAID-массива
  • vg0.cfg - таблица разметки для менеджера логических томов LVM
  • pv_uuid - уникальный идентификатор физического тома LVM
  • data_uuid - уникальный идентификатор логического тома DATA
  • system.img.gz - сжатый образ логического тома SYSTEM
Также в некоторых конфигурациях менеджер загрузки может быть вынесени на отдельный диск или раздел диска. В этом случае создается сжатый образ этого диска, который называется boot.img.gz

Эти файлы содержат копию структуры жесткого диска и копию операционной системы, со всеми настройками. После создания такой резервной копии, вы в любой момент можете восстановить систему в состояние, в котором она была на момент создания этой копии. Вы можете привести систему в это состояние даже если данные жесткого диска будут уничтожены полностью.

Восстановление системы из резервной копии

Если текущее состояние системы не устраивает вас, можно вернуть ее в то состояние, в котором она работала хорошо, при условии что у вас есть резервная копия этого состояния.

Чтобы восстановить систему из резервной копии нужно загрузить сервер с CD-диска аварийного восстановления и подключить носитель, на котором есть папка с нужной резервной копией системы.
При восстановлении из резервной копии, с жесткого диска стирается все, кроме раздела DATA и затем всё это восстанавливается из резервной копии. Процедура восстановления гарантирует сохранность данных, находящихся на разделе DATA при условии, что таблица разметки диска и таблица разметки менеджера томов LVM не менялись с момента создания резервной копии, которую вы хотите использовать для восстановления.
Несмотря на эти гарантии, мы рекомендуем перед процедурой аварийного восстановления, войти в папку "Данные разделов" на рабочем столе и сделать резервную копию всех критически важных данных (если разрушения системы позволяют вам видеть данные разделов).
Перед восстановлением системы из резервной копии нужно убедиться, что RAID-массив находится в целостном состоянии. Для этого откройте Web-интерфейс Webmin и войдите в раздел "Оборудование" --> "Linux RAID". Если RAID-массив находится в состоянии degraded, приведите его в состояние "clean" или "active". После приведения RAID-массива в рабочее состояние снова перезагрузите сервер с CD-диска аварийного восстановления.

Для восстановления системы из резервной копии нужно:

1. Загрузить программу "Восстановление системы" с рабочего стола
 
2. Выполнить команду "Восстановить систему из резервной копии" из меню



3. Найти устройство, на котором находистся резервная копия, найти на нем папку, в которой она находится, выбрать эту папку (не входить в нее) и нажать ОК.



4. Начнется процесс восстановления системы из резервной копии. Для сохранения производительноти, мы никак не визуализируем этот процесс. Он будет идти незаметно. Вам нужно просто дождаться завершения процедуры восстановления системы из резервной копии, о чем сигнализирует сообщение "Восстановление системы из резервной копии успешно завершено". Обычно процедура длится не более 15 минут.

Процедура восстанавливает не только систему в состояние, в котором она была на момент создания резервной копии, но и загрузочную запись и таблицу разметки дисков. Поэтому, если сбой был связан с загрузочным сектором и вы потеряли доступ к данным на разделе DATA, вполне вероятно что после восстановления загрузочного сектора эта проблема решится и раздел DATA станет доступен. Сразу после восстановления системы проверьте это: откройте папку "Данные разделов" с рабочего стола и войдите в папку "DATA". В ней должно быть содержимое раздела данных.

Если проблема была серьезнее и данные были стерты, то раздел DATA будет после восстановления пустым. В этом случае может спасти только резервная копия данных.

Утилита "Управление логическими томами"

С помощью этой утилиты вы можете изменять размеры разделов жесткого диска.



На изображении справа схематично показано логическое представление диска, как его видит LVM и физическое представление диска, которое было образовано утилитой fdisk. Физически жесткий диск состоит из одного раздела /dev/md127 - это устройство RAID. Логически, он разбит на три раздела - DATA, SWAP и SYSTEM. Слева в списке развернута группа "Логический вид" и мы можем выбирать любой из логических разделов и редактировать его свойства.
Одно из свойств раздела это его размер. Можно увеличивать и уменьшать размер каждого раздела. При этом утилита также автоматически увеличивает и уменьшает размер файловой системы, которая находится на этом разделе. Утилиты LVM командной строки этого не делают, поэтому из командной строки это выполняется в два этапа: сначала увеличивают размер логического тома, а затем увеличивают размер файловой системы. Если нужно уменьшить размер, то наоборот: сначала уменьшают размер файловой системы, а потом размер логического тома, причем, для уменьшения файловой системы требуется ее отмонтировать. Именно поэтому мы можем изменять размеры разделов только с диска аварийного восстановления системы, потому что иначе не получится отмонтировать системный раздел.
Утилита выполняет всё это за нас автоматически, однако это не меняет суть операций, которые выполняются. Увеличить раздел жесткого диска можно за счет свободного пространства, которое не относится ни к одному и разделов. Однако на сервере все пространство занято логическими томами. Поэтому, прежде чем увеличить размер одного тома, нужно сначала уменьшить размер другого. Перед уменьшением размера, файловую систему этого тома нужно отмонтировать, а при работающем сервере она скорее всего будет занята и отмонтировать ее не получится, особенно если речь идет о томе SYSTEM. Поэтому наиболее безопасно выполнять эти операции с диска аварийного восстановления, когда система полностью остановлена (несмотря  на то, что LVM позволяет увеличивать размер раздела на лету, без остановки файловой системы).
Предположим, нам нужно увеличить размер раздела SYSTEM на 2 Гб. Для этого  нужно уменьшить размер раздела DATA на 2 Гб и только потом увеличить раздел SYSTEM за счет освободившегося пространства. Естественно нужно быть уверенным, что на разделе DATA есть 2 Гб свободного пространства. Выберите раздел DATA в списке слева и нажмите кнопку "Изменить свойства" внизу экрана.


В поле "Размер LV" уменьшим размер так, чтобы он стал на 2 Гб меньше. В данном примере получится число 16.48 и нажмем кнопку "ОК".
Теперь аналогично откроем свойства раздела "SYSTEM":


Здесь видна надпись: "Осталось места для этого Тома 2.0 Гигабайт". Таким образом мы можем использовать освободившееся пространство для увеличения размера этого тома. Теперь можно либо ввести новый размер в поле "Размер LV", либо нажать кнопку "Использовать оставшееся" и все свободное пространство будет добавлено к размеру раздела. После этого нажмите "OK".

Все изменения применяются сразу, утилита выполняет все необходимые операции по отмонтированию и монтированию файловых систем. Поэтому вы сразу можете продолжать работать с разделами новых размеров, без дополнительных действий.

Создание резервных копий данных сервера

Данные, расположенные на сервере в папке /data тоже необходимо резервировать.

Основы резервного копирования данных

Данные, в отличии от системы постоянно меняются, поэтому стратегию резервного копирования и восстановления данных выбрать не так просто. Этот выбор зависит от многих факторов, таких как общий объем данных, объем постоянно изменяющихся данных, размер файлов, количество доступного пространства для создания резервных копий, скорость носителя с резервными копиями, а также скорость и надежность связи с хранилищем резервных копий, если резервные копии сбрасываются во внешнее хранилище. Для того чтобы покрыть большинство вариантов, LVA Business Server предлагает технические средства для реализации нескольких стратегий резервного копирования и восстановления данных. Эти средства основаны на двух классических схемах резервного копирования, которые в оригинале называются полный бэкап (full backup) и инкрементный бэкап (incremental backup). На базе этих методов появились другие, такие как методы создания резервных копий, основанных на  моментальных снимках состояния данных.

Полный бэкап

Как говорит само название, полный бэкап предназначен для создания полных резервных копий данных. По этой схеме создается архив с полной резервной копией всех или части данных. Обычно в название архива или папки с архивом включают дату и время создания резервной копии. Архив сжимают и переносят в хранилище резервных копий, которое может физически размещаться либо на самом сервере в определенной папке, либо на внешнем носителе или на отдельном архивном сервере, доступ к которому осуществляется по сети. В качестве архива может выступать либо раздел на жестком диске, папка на диске, либо сжатый файл.

Обычно сервер настраивают так, чтобы он создавал резервные копии по расписанию, например раз в сутки ночью.



В дальнейшем, для восстановления файла из резервной копии, администратору нужно открыть архив за нужное число и извлечь из него нужный файл. Для восстановления всего раздела данных, достаточно развернуть весь архив целиком поверх раздела данных.

К плюсам полного бэкапа можно отнести его простоту и надежность. Имея архив за определенное число, администратор имеет все данные на это число и может восстановить раздел данных полностью, не обращаясь к каким-либо другим архивам.

К минусам полного бэкапа можно отнести нерациональность использования дискового пространства и системных ресурсов. Каждый архив содержит все данные, независимо от того, изменялись они или нет, поэтому в системах, в которых данные меняются редко, полный бэкап избыточен. Кроме того, на создание полного бэкапа требуется значительно больше времени и ресурсов процессора, поэтому скорее всего не получится делать его слишком часто. Обычно полный бэкап делают ночью или в другое время, когда пользователи не работают с данными.

Инкрементный бэкап

Альтернативой полному бэкапу является бэкап инкрементный. Суть инкрементного бэкапа заключается в том, что в архив попадают только файлы, которые изменились с момента создания предыдущего архива.

Для реализации стратегии инкрементного бэкапа, первоначально создается полный архив, а затем архивы, в которых находятся изменившиеся по отношению к полному архиву файлы.. Программа, создающая инкрементный архив сравнивает текущие данные с данными, находящимися в полном архиве и включает в создаваемый архив только такие файлы, которые отличаются. В следующий раз, эта программа сравнивает текущие данные с данными, находящимися в полном бэкапе и в предыдущих инкрементных и т.д.



Судя по изображению, сначала утилита резервного копирования создала полный архив за 12.05.2012 г. Затем, на следующий день изменился только файл "file1.doc", поэтому утилита сделала инкрементный архив, в который вошел только этот файл. Затем, 14.05.2012 изменился только файл file2.zip. Именно он и попал в архив за это число.

Для восстановления файла в системе, в которой применяется инкрементный бэкап, системному администратору нужно найти самый последний архив, в котором этот файл изменился и извлечь этот файл из него. Чтобы восстановить раздел данных целиком в состояние на определенное число, нужно сначала развернуть полный архив, а затем последовательно все инкрементные архивы, которые были сделаны после полного, вплоть до нужного числа.

К плюсам инкрементного бэкапа можно отнести его экономичность по сравнению с полным. Архивы с разницей создаются значительно быстрее и занимают значительно меньше пространства на диске, что позволяет делать их чаще и таким образом иметь в архиве более детальную историю изменения данных.

К минусам инкрементного бэкапа можно отнести его относительные неудобства при восстановлении данных. Имея инкрементный архив, администратор не может судить в целом о состоянии данных на дату этого архива не прибегая к другим инкрементным архивам, которые были созданы раньше или позже. Администратор не может точно знать, является ли файл в конкретном инкрементном архиве самой последней его версией. Также, для восстановления данных из архива целиком, например на новый жесткий диск, требуется последовательно распаковывать все архивы, начиная от впервые созданного полного и до самого последнего инкрементного.

Бэкап с помощью моментальных снимков

Особенностью классических методов полного и инкрементного бэкапа является тот факт, что для архивирования с помощью этих методов требуется иметь как минимум столько же дискового пространства, сколько занимают все архивируемые данные. Если применяется полный бэкап, то требуется в десятки раз больше дискового пространства. Если применяется инкрементный бэкап, требуется изначально столько же пространства, сколько занимают все данные, а затем дополнительное пространство для создания инкрементных архивов. Фактически для архивирования нужен еще один такой же сервер и это как минимум. А что же делать, если его нет, но при этом нужно архивировать изменяющиеся данные и иметь возможность восстановить их из предыдущей версии ?

В качестве попытки решения этой проблемы в наших продуктах предлагается технология моментальных снимков данных. Моментальный снимок это раздел жесткого диска, точная копия раздела данных. Отличительная особенность этого раздела заключается в том, что этот раздел является виртуальным и вместо файлов и папок, на нем находятся ссылки на эти файлы и папки, то есть изначально данные на этом разделе не занимают дискового пространства, но при этом он отражает состояние данных на определенный момент времени и может быть основной для создания инкрементных архивов, то есть, заменить полный архив. Поэтому размер такого раздела может быть значительно меньше, чем размер раздела с данными.

При создании, снимок, так же как и раздел данных монтируется в файловую систему (по умолчанию в папку /data/snapshots) и в дальнейшем он тесно связан с разделом данных. Как только на разделе данных меняется файл, система реагирует на это, заменяя на снимке ссылку этого файла на резервную копию этого файла. Таким образом, снимок становится инкрементным архивом, который заполняется постепенно, по мере изменения данных. Снимок может заполняться данными до тех пор, пока эти данные не достигнут размера, изначально выделенного для этого снимка, поэтому снимок обычно может содержать резервную копию лишь части данных, а не все данные раздела. Администратору нужно предугадать, сколько могут занимать данные, которые изменятся на сервере и указать это число в качестве размера снимка, при его создании. Если создавать такие снимки по такому же принципу, как архивы, то есть по расписанию, нужно выделять каждому снимку  количество дискового пространства, достаточное для того, чтобы он не успевал заполняться полностью до момента создания следующего снимка.



На рисунке показана организация резервного копирования на базе моментальных снимков. Первое, что заметно сразу, здесь нет внешнего хранилища архивов. Моментальный снимок это некая виртуальная конструкция, запускаемая внутри операционной системы. Снимки тесно связаны с разделом данных и могут работать только в одной операционной системе с ним и не будут работать во внешнем хранилище. Мы видим три снимка, для каждого из них выделено по 10 Гб дискового пространства. Самый последний из них пуст, все файлы на нем затенены, то есть, являются ссылками на файлы, находящиеся на самом разделе данных, а с точки зрения конечного пользователя, эти ссылки являются точными копиями самих файлов.

На предыдущем снимке мы видим что один файл (file1.doc) "проявился". Это значит, что 13 числа этот файл был изменен в разделе данных и на снимке "проявилась" его резервная копия по состоянию на 13 число до того изменения. Так как все три снимка работают одновременно и независимо друг от друга, мы также видим, что этот файл проявился и на снимке от 12.05.2012. В обоих снимках получился одинаковый файл, так как и на начало 12 числа и на начало 13 числа он был одинаков и только затем изменился. Также мы видим что 12 числа изменился файл file2.zip ,его резервная копия проявилась только на снимке от 12 числа.Таким образом снимки постепенно заполняются, по мере "проявления" на них резервных копий изменяющихся файлов. Заполняться они могут вплоть до 10 Гб. Как только снимок заполняется на 100%, он деактивируется и становится недоступен. Все данные на снимке теряются и восстановить их невозможно, можно только удалить снимок. Следовательно нельзя допускать полного заполнения снимков, нужно следить за этим и увеличивать их размер по мере необходимости. Операционная система и средства администрирования содержат все необходимое для этого.

Для восстановления файла из резервной копии, находящейся в снимке, системном администратору нужно просто войти в каталог, к которому примонтирован снимок, найти файл и восстановить его. Для того чтобы восстановить данные из резервной копии полностью, достаточно развернуть содержимое снимка поверх содержимого раздела данных.

К плюсам резервного копирования с использованием моментальных снимков можно отнести его экономичность и скорость. С помощью моментального снимка администратор получает все преимущества полного бэкапа по цене более низкой, чем инкрементный. Снимок отображает весь раздел целиком, поэтому администратор, используя один снимок, может видеть полностью состояние раздела данных на момент создания снимка. Ему не нужно, как в случае с инкрементным бэкапом "собирать его по кусочкам". Снимок фактически является полным бэкапом состояния раздела, но на создание снимка уходит доли секунды, поэтому снимки можно создавать в любой момент, эта операция не создает никакой нагрузки.

Основной минус моментального снимка заключается в том, что это очень хрупкая виртуальная конструкция, созданная внутри операционной системы и жестко привязанная к разделу данных, его нельзя рамещать во внешнем хранилище. Снимок рушится как только заполняется на 100%.  Также, операционная система видит снимок как полноценный раздел и показывает все его данные, в ней не существует стандартных средств, позволяющих отделить виртуальные ссылки от реальных резервных копий файлов, которые "проявились" на снимке. В отличии от инкрементных архивов, которые создаются и копируются последовательно, все созданные снимки активны и работают одновременно. Поэтому, если на разделе данных изменяется файл, который ранее не менялся, он "проявляется" и хранится сразу во всех снимках (а не только в последнем, как это было бы при использовании инкрементного резервного копирования). Таким образом в разных резервных копиях могут накапливаться одинаковые файлы, занимающие дисковое пространство.

Инкрементный бэкап с помощью моментальных снимков

Мы синтезировали описанные выше стратегии и взяли лучшее из них, получив таким образом классический инкрементный бэкап, который в качестве базы использует не полный архив раздела данных, а его моментальный снимок. Таким образом получилась довольно оптимальная система резервного копирования, лишенная почти всех недостатков, которые были у предыдущих.



В соответствии с этой схемой, на сервере выполняется ротация снимка, то есть с определенной периодичностью выполняется одна операция, описанная далее.

Сервер создает один снимок раздела данных на определенный момент времени. Затем этот снимок по мере изменения данных заполняется резервными копиями измененных или удаленных файлов. Далее запускается утилита и копирует все содержимое, которое накопилось в снимке в папку во внешнем хранилище, создавая таким образом инкрементный архив. Затем этот снимок удаляется и на его месте создается новый снимок, отражающий текущее, новое  состояние раздела данных. Затем все повторяется снова и снова. Таким образом в хранилище накапливаются инкрементные архивы, в которых находятся только изменившиеся файлы.

Если судить по изображению, сначала был создан снимок 12-го мая, затем в течении дня в нем проявилась резервная копия файла file2.zip и содержимое этого снимка было сброшено в хранилище в качестве инкрементного архива. Затем на месте снимка за 12-е число был создан снимок за 13-е число. В нем в течении дня проявилась резервная копия файла file1.doc и была сброшена в хранилище в инкрементный архив за 13.05.2012. Снова снимок был удален и вместо него создан новый снимок за 14 мая. Затем в течении дня снова изменился файл file2.zip и его резервная копия проявилась на снимке. Наконец содержимое снимка (с одним лишь файлом) было сброшено в хранилище архивов как инкрементный архив за 14.05.2012 и снимок снова обновился. Таким образом у нас появилась история изменений за 3 дня. file1.doc изменился один раз, а файл file2.zip изменился дважды.

К плюсам этой стратегии можно отнести наибольшую экономичность, скорость и производительность резервного копирования. В качестве базы для создания инкрементных архивов используется моментальный снимок, что не требует никакого дискового пространства, его создание не создает нагрузки, но при этом он выполняет функцию полного архива, от которого отталкивается система создания инкрементных архивов. Кроме того, снимок всего один, поэтому система не тратит ресурсы на поддержку одновременно нескольких снимков. Если изначально правильно подобрать его максимальный размер (что значительно проще сделать, если он всего один), то он не переполнится до следующей ротации.

К минусам, пожалуй, можно в определенных случаях отнести отсутствие возможности восстановить раздел данных с нуля из хранилища архивов, потому что нет полного архива, на базе которого создаются инкрементные. Однако такое может потребоваться только при полном выходе из строя сервера. Вторым недостатком можно посчитать нагрузку, которая создается при копировании содержимого снимка во внешнее хранилище. Как ранее было описано, в операционной системе не существует средств, позволяющих выявить различия между разделом данных и снимком, система видит снимок как полноценный раздел и при запуске простого копирования будет копировать не только проявившиеся резервные копии, но и все остальные неизмененные файлы. Для того чтобы выявить и скопировать только отличающиеся файлы, запускается процедура пофайлового сравнения снимка и раздела данных с использованием контрольных сумм, вычисление которых требует процессорного времени.

Выбор стратегии резервного копирования

LVA Business Server напрямую поддерживает три из четырех описанных стратегий:
  1. Полный бэкап
  2. Бэкап с помощью моментальных снимков
  3. Инкрементный бэкап с помощью моментального снимка
При необходимости можно также имитировать классический инкрементный бэкап,  совместив 1 и 3, то есть, сделать полный бэкап и запустить систему создания инкрементных архивов с помощью моментального снимка.

Если в сервере нет RAID-массива, то нужно учитывать вероятность выхода из строя жесткого диска с последующей его заменой. Поэтому должна быть возможность полного восстановления раздела данных. В этой ситуации может помочь только полный бэкап или сочетание вариантов 1 и 3, как описано выше.

Если вы хотите хранить историю изменений файлов, чтобы была возможность восстановить файл в любое состояние, то наиболее экономичным будет выбор между вариантами 2 и 3. В этом случае выбор зависит от того, насколько долго должны храниться разные версии каждого файла и насколько много их должно храниться. Если достаточно всего нескольких дней, то можно запустить бэкап с помощью моментальных снимков прямо на самом сервере (вариант 2), создать столько снимков, сколько резервных копий должно храниться и запустить ротацию. При этом по мере заполнения снимков, устаревающие будут автоматически удаляться, а новые появляться. Такой метод будет ограничен дисковым пространством самого сервера (обычно мы оставляем около 200 Гб для снимков) и количеством созданных снимков. Если у вас большой объем изменяемых данных, то снимки будут "ротироваться" довольно быстро и соответственно, прошлые версии файлов будут исчезать часто.

Если у вас есть внешнее хранилище для архивов, то прошлый вариант можно улучшить и при ротации сбрасывать содержимое устаревающих снимков в это хранилище в виде инкрементных архивов. Также, в этом случае вы полностью можете перейти на 3-й вариант, то есть оставить в ротации всего один снимок и сделать чтобы после каждой ротации его содержимое сбрасывалось в хранилище. Возможно прежде чем определиться с тем, один снимок оставлять или больше, нужно поэкспериментировать. Дело в том что данные из снимков будут читаться быстрее, чем из резервных копий, хранящихся во внешнем хранилище и если требуется частый доступ к прошлым версиям, можно сделать чтобы несколько последних прошлых версий хранилось в снимках, а остальные в хранилище, в виде инкрементных архивов. Подобрать наиболее оптимальный вариант можно только опытным путем.

Наша система содержит средства управления резервным копированием данных, позволяющие реализовать все описанные выше стратегии. Для реализации полного бэкапа используется интерфейс Webmin. Здесь используется стандартная технология, которую мы не видим смысла каким-либо образом модифицировать. Работа с ней описана далее в разделе "Подсистема архивирования данных".

Работа с моментальными снимками является основой для подсистемы, получившей название "Теневое копирование данных", работа с которой описана в соответствующем разделе этого руководства. Эта подсистема реализована в нашей панели управления, а также, наш сервер обеспечивает совместимость моментальных снимков и инкрементных архивов, находящихся во внешнем хранилище с интерфейсом ShadowCopy. Этот интерфейс поддерживает система Windows, с его помощью обычный пользователь может стандартными средствами получить список версий каждого файла, на основании накопившихся в разных местах (в снимках и в архивах) резервных копий и с помощью проводника Windows построить состояние раздела данных или любой его папки на любой момент времени, в который была сделана резервная копия на сервере.

Backup options



Параметры этого раздела позволяют настроить параметры группы "как копировать". Кратко перечислим значения этих параметров, однако скорее всего изменять их не придется, так как их значения для большинства случаев оптимальны.
  • Метка резервной копии - в случае если копирование производилось бы на ленточный накопитель, то можно установить метку этой резервной копии, чтобы в дальнейшем ее можно было быстрее найти на ленте
  • Размер ленты - указывается размер ленты, в случае если резервная копия создается на ленточном накопителей
  • Files and directories to skip - можно указать через пробел список файлов и каталогов, которые не нужно помещать в резервную копию
  • Compress archive ? - указывается, каким архиватором сжимать архив: gzip, bzip2 или не сжимать
  • Разбивать на несколько файлов - стоит ли разбивать бэкап на несколько файлов. Эта опция вычисляет максимально допустимый размер файла для файловой системы, в которой размещаются резервные копии и разбивает резервную копию на файлы этого размера
  • Follow symbolic links ? - если "Да", то при копировании символьных ссылок будут копироваться не ссылки, а файлы, на которые эти ссылки ссылаются, иначе будут копироваться сами ссылки
  • Limit backup to one filesystem ? - если "Да", то будут копироваться только файлы, находящиеся в пределах одной файловой системы. Если внутри папок, назначенных для резервирования будут примонтированы папки с данными в других файловых системах, они копироваться не будут
  • Prompt for new tape if needed ? - если "Да", то если на носителе заканчивается свободное место в процессе создания резервной копии, просит "вставить новый носитель" для продолжения копирования
  • Just add new files to archive ? - обычно, при создании резервной копии, предыдущая резервная копия заменяется. Если этот параметр установлен в "Да", то этого происходить не будет. В этом случае просто будут добавляться новые и измененные файлы в текущую резервную копию
  • Remount with noatime option - Если "Да", то перед началом резервного копирования файловая система с данными для копирования будет перемонтирована с параметром noatime. Параметр noatime говорит файловой системе не записывать каждый раз время доступа к файлам файловой системы. Этот параметр может ускорить процесс копирования
  • Path to rmt on remote system - при создании резервных копий с помощью TAR, на удаленном сервере вызывается команда rmt. В этом параметре можно указать путь к этой команде, если он отличается от пути по умолчанию
  • Attempt test restore after backup to verify - если "Да", то после резервного копирования будет предпринята попытка проверить корректность операции, перечислив файлы, входящие в резервную копию. Если эта команда сбойнет, то резервная копия будет считаться "битой", что будет показано на экране и записано в журнал.
  • Extra command-line parameters - в этом поле можно указать дополнительные параметры команды tar, которые будут использоваться при копировании
  • Команда, выполняемая перед копированием - здесь можно указать произвольную команду или последовательность команд Bash, которые будут выполняться перед созданием резервной копии. Это может быть очистка, удаление устаревших резервных копий и т.п.
  • Команда выполняемая после копирования - здесь можно указать произвольную команду или последовательность команд Bash, которые будут выполняться сразу полсе создания резервной копии. В некоторых конфигурациях это поле по умолчанию заполнено. В нем указана команда: mv /data/share/backup/backup.tar.gz "/data/share/backup/`date +"%d_%m_%Y"`.tar.gz" . Эта команда сразу после создания резервной копии в файле backup.tar.gz переименовывает этот файл. В качестве имени файла используется текущая дата. Это позволяет не перезаписывать каждый раз файл backup.tar.gz, а создавать каждый день новый файл, содержащий резервную копию данных, актуальную на указанную дату.

Подсистема теневого копирования данных

Этот вид резервного копирования, в отличии от простого архивирования, не требует прямого вмешательства системного администратора в процесс создания резервных копий. Он основан на использовании моментальных снимков состояния данных. Теоретические основы резервного копирования в целом и в частности основы резервного копирования с использованием моментальных снимков представлены  в разделе "Основы резервного копирования данных", который мы настоятельно рекомендуем прочитать и понять до того, как читать дальше эту тему.

Если на определенный момент времени создан моментальный снимок раздела данных, резервные копии файлов раздела на этот момент времени будут создаваться в снимке автоматически в момент их изменения в разделе данных. Такое копирование называется теневым, потому что оно происходит "в тени", или можно сказать, "за кулисами". Системному администратору не требуется выделять для него отдельное время или останавливать работу.

Основные параметры теневого копирования

В процессе работы механизма теневого копирования создаются снимки состояния раздела данных. Раздел, снимок которого создается находится в группе томов LVM, имя которой указано в поле "Группа разделов данных". Имя самого раздела указывается в поле "Раздел данных". Снимок является таким же разделом, как и раздел данных, поэтому он должен монтироваться в файловую систему. Снимки монтируются в папку, указанную в поле "Хранилище снимков". При создании снимка, в этом каталоге создается каталог с названием в формате "@GMT-<дата-и-время-создания-снимка>" и в него монтируется сам снимок. Название каталога используется подсистемой ShadowCopy, встроенной в Windows для того чтобы определять, что этот каталог является хранилищем резервных копий файлов на указанное в названии этого каталога время.

В разделе "Основы резервного копирования данных" и в частности в его подразделе "Бэкап с использованием моментальных снимков" говорилось о том, что нельзя допускать 100%-го заполнения снимка, нужно следить чтобы он не переполнялся, при необходимости и возможности увеличивать его размер или успеть сбросить его данные в папку на диске и удалить. Эти операции можно выполнять вручную, для чего предназначена закладка "Снимки" данного окна, а можно настроить систему так, чтобы она делала это автоматически, используя следующие поля.

Система может автоматически увеличивать размер снимка, если его заполненность превышает процент, указанный в поле "Критический процент заполненности снимка (%)". Размер снимка в этом случае будет увеличиваться на число, указанное в поле "Размер приращения снимка при достижении критической заполненности (Гб)". Если эти два поля заполнены, каждые 5 минут на сервере запускается утилита, которая проверяет процент заполненности активных снимков и пытается увеличить их при необходимости. Если у нее не получается это сделать по причине отсутствия на диске достаточного пространства, она автоматически удаляет самый старый снимок, освобождая таким образом пространство на диске и пытается снова. В случае если заполнено поле "Сбрасывать снимок в каталог при автоматическом удалении", то перед удалением снимка, его содержимое сбрасывается в каталог, указанный в этом поле. Внутри этого каталога создается каталог в формате "@GMT-<дата-и-время-создания-снимка>" и в него сбрасываются резервные копии всех файлов, которые изменились с момента создания этого снимка, с сохранением структуры папок.

В некоторых конфигурациях сервера может быть предустановлен специальный интерфейс, предназначенный для просмотра резервных копий, сделанных подсистемой теневого копирования прямо в браузере. Этот интерфейс не зависит от Панели управления сервером, запускается отдельно и доступен каждому пользователю. Также, этот интерфейс доступен на закладке "Резервные копии" в окне свойств файла или папки, если открыть это окно с помощью Файлового менеджера. В поле "Адрес интерфейса для просмотра резервных копий" указывается путь к этому интерфейсу.

После установки основных параметров, система запустится и вам нужно определиться, каким образом вы организуете сам процесс теневого резервного копирования. Вы можете создавать снимки вручную и получать резервные копии измененных данных именно на тот момент, который вам нужен и вручную сбрасывать данные снимков в архивы при заполнении снимков, либо вы можете настроить систему так, чтобы она делала это автоматически. В зависимости от выбранного варианта, вам нужно использовать закладки "Снимки" и "Ротация".

На закладке "Снимки" находится список созданных моментальных снимков.



Таблица отображает состояние снимков на момент открытия окна. Чтобы получить текущую информацию, нужно "Обновить таблицу", нажав соответствующую кнопку над ней. Вам нужно следить за заполненностью снимков (колонка "Заполненность (Гб)"). Если заполненность равна 100%, снимок деактивируется и в поле "Активность" его состояние становится равно "Не активен". Единственное что можно сделать со снимком, который перешел в это состояние, это выделить его, щелкнув по нему мышью и удалить с помощью кнопки над таблицей. Чтобы такого не произошло, нужно вовремя увеличить размер снимка (или настроить автоматическое его увеличение на закладке "Параметры" этого окна). Для того чтобы увеличить размер снимка вручную, нужно выделить строку со снимком, нажать кнопку ("Изменить размер"), ввести новый размер и нажать кнопку "ОК". Также изменить размер можно просто сделав двойной щелчок по строке со снимком. При установке размера, нужно учитывать значение в строке "Свободно для снимков", которая находится под таблицей. Нельзя указывать размер больший, чем указан в этой строке.

В большинстве случаев сервер настраивают таким образом, чтобы новые снимки создавались автоматически в определенное время по расписанию, а устаревающие также автоматически удалялись. Этот процесс называется ротацией снимков и настраивается на закладке "Ротация". Однако иногда вам может понадобиться сохранить данные в текущем состоянии именно сейчас. В этом случае можно создать снимок вручную. Для этого нужно нажать кнопку , ввести размер создаваемого снимка и нажать "ОК". После этого вы можете быть уверены, что все данные, которые с этого момента будут меняться сохранятся в данном снимке именно в том состоянии, в котором они находятся в данный момент.

По мере изменения данных, снимок заполняется и вы можете посмотреть его содержимое, выделив строку со снимком и нажав кнопку ("Просмотреть содержимое снимка"). В новом окне появится список резервных копий файлов, содержащихся в снимке. Если файлов очень много, процедура формирования этого списка может занять длительное время.



В самой верхней строке указан каталог, к которому примонтирован раздел-снимок. Затем идет список полных путей к файлам, содержащихся в снимках. Каждая строка является гиперссылкой, нажав по которой вы можете загрузить резервную копию этого файла, сохраненную в снимке. Таким образом можно не прибегая к средствам операционной системы восстанавливать файлы из резервной копии. Также, после имени файла, в скобках есть гиперссылка "Текущее состояние файла", нажав на которую вы можете загрузить этот файл из раздела данных, в текущем состоянии, и сравнить, чем он отличается от своей резервной копии, сохраненной в снимке. Если при нажатии на эту ссылку браузер показывает "File not found", значит у файла нет текущего состояния, то есть он был удален, а его резервная копия сохранилась в этом снимке и вы можете загрузить ее на свой компьютер. Таким образом можно восстанавливать ранее удаленные файлы.

Также, вместо восстановления файлов по отдельности, можно выгрузить все содержимое снимка целиком. Выделив строку и снужным снимком и нажав на кнопку над таблицей, вы можете выгрузить его содержимое в папку на диске сервера. Вам нужно будет выбрать эту папку. После выбора, в этой папке будет создана папка снимка в формате "@GMT-<дата-и-время-создания-снимка>" и в нее будет записано содержимое снимка. Эта папка будет инкрементным архивом раздела данных на указанное в ее названии время. Если в выбранной папке уже есть такая папка снимка, она будет удалена и создана заново. Если объем данных большой, их копирование может занять длительное время.

Ротация снимков

Вы можете вручную управлять созданием снимков, их удалением и выгрузкой данных из них, однако скорее всего будет удобнее автоматизировать этот процесс, что делается на закладке "Ротация". Чтобы включить ротацию, нужно установить флажок "Включить ротацию снимков".



На этой закладке указывается в полях "Дни", "Часы" и "Минуты" с какой периодичностью будут создаваться снимки, какого размера будет каждый новый снимок (поле "Размер нового снимка") и какое количество их будет в ротации (поле "Количество снимков в ротации").

После заполнения этих полей формы и нажатия кнопки "Сохранить", ротация начнет работать. Алгоритм ее работы следующий:

С указанной периодичностью будет запускаться утилита, которая будет создавать новый снимок указанного размера и добавлять его в список созданных снимков. Как только количество снимков в этом списке превысит значение в поле "Количество снимков в ротации", утилита автоматически удалит самый старый снимок. Эта операция будет считаться "автоматическим удалением", поэтому если на закладке "Настройки" заполнено поле "Сбрасывать снимок в каталог при автоматическом удалении", то перед удалением, содержимое снимка будет копироваться в каталог, указанный в этом поле, при этом в указанном каталоге будет создан подкаталог с именем в формате "@GMT-<дата-и-время-создания-снимка>" и в него будут записаны все файлы, содержащиеся в снимке с сохранением структуры папок.

Если не указана папка, в которую копируются устаревающие снимки, то реализуется стратегия "Бэкап с помощью моментальных снимков", при которой несколько снимков будут содержать резервные копии файлов и эти копии будут автоматически удаляться вместе с устаревающими снимками. В результате получится система резервного копирования, работающая полностью автоматически. Если же папка, в которую копируются устаревающие снимки указана и в качестве значения поля "Количество снимков в ротации" указано значение "1", то реализуется стратегия "Инкрементный бэкап с помощью моментальных снимков". В этом случае при каждой ротации содержимое этого единственного снимка будет копироваться в архив, в указанную папку, снимок будет удаляться и вместо него будет создаваться новый. Это позволяет хранить неограниченное количество резервных копий файлов в архиве, но необходимо следить, чтобы архив не переполнился.

Периодичность выполнения ротации задается в полях "Дни", "Часы" и "Минуты" по такому же принципу, как она задается при создании расписания резервного копирования, как было описано в разделе "Подсистема архивирования данных". В каждом из этих полей может указываться то, что понимает стандартный планировщик задач cron. Это может быть число, список чисел, разделенных запятыми, диапазон чисел, разделенных символом "-", символ "*", символ "/" или их комбинация.

Если, например в поле "Минуты" указано просто число, это означает, что действие будет выполняться только в эту минуту часа. Если указан список чисел, через запятую, то действие будет выполняться в каждую из указанных минут часа. Также может быть указан диапазон чисел, например 2-10. В этом случае действие будет выполняться в каждую минуту из диапазона. Если указан символ "*", значит действие будет выполняться каждую минуту часа. Также можно после любого из описанных выше вариантов указывать символ "/" и число. Этот символ в данном случае переводится как "через". Например если указано */2, то читается это следующим образом: "каждую минуту через две минуты", то есть "каждые две минуты", или если указано 2-10/2, то это значит, что действие будет выполняться со второй по десятую минуту через две минуты. Это равносильно последовательности 2,4,6,8,10.

Аналогичное указывается и в полях Дни и Часы. А теперь попробуем объединить все вместе и понять, с какой периодичностью будет выполняться действие, если его периодичность задана так, как она задана на прошлом изображении. Итак, "*" означает "каждый день", далее "*/6" в поле часы означает "каждый час через шесть часов" или "каждые шесть часов", и в поле "Минуты" установлено "1", то есть в первую минуту часа. Объединив все получаем: утилита будет выполняться ежедневно каждые шесть часов в первую минуту часа.

Расписание, созданное на основании значений, указанных в этих полях записывается в конфигурационный файл cron (/etc/crontab). Подробнее о создании расписаний для cron можно прочитать на man-странице по этому конфигурационному файлу. Чтобы прочитать ее, нужно подключиться к серверу в режиме командной строки и выполнить команду:

man 5 crontab

Чтобы отключить ротацию снимков, нужно снять флажок "Включить ротацию снимков" и нажать "Сохранить".

Чтобы полностью отключить механизм теневого копирования данных, нужно убрать флажок "Включить теневое копирование данных" и нажать кнопку "Сохранить".

ВНИМАНИЕ: При отключении теневого копирования данных, все активные снимки, которые можно видеть в таблице на закладке "Снимки" удаляются, вместе с их содержимым (снимки, сброшенные в архив остаются нетронутыми).

Работа с резервными копиями

Отличительной особенностью подсистемы теневых копий является удобство работы пользователя с этим механизмом. Чтобы восстановить файл из резервной копии, пользователю не нужно приглашать системного администратора и ожидать, пока он будет "рыться в архивах" и искать нужный файл. Функциональность теневого копирования встроена в любую версию системы Windows.

В стандартных дистрибутивах операционной системы GNU/Linux такая возможность не предусмотрена, но если на вашем компьютере установлена система LVA Business Desktop, основанная на GNU/Linux, то в ней такая возможность есть. Она интегрирована в стандартный файловый менеджер и работает так же, как в Windows.


Если пользователь понял, что испортил файл и хочет восстановить его из одной из прошлых копий, ему нужно просто нажать правой кнопкой мыши по этому файлу и выполнить команду "Свойства" из контекстного меню.



Если на сервере включено теневое копирование данных, в окне свойств появится закладка "Предыдущие версии", на которой будут показаны все предыдущие версии данного файла, сохраненные в снимках и в инкрементных архивах на сервере (в папке, указанной в поле "Сбрасывать снимок в каталог при автоматическом удалении" в окне "Теневые копии" на закладке "Параметры") с соответствующими датами и временем создания этих снимков и икрементных архивов. Пользователь может с помощью соответствующих кнопок "Просмотреть" любую версию файла, "Копировать" файл этой версии в папку или "Восстановить" файл из этой версии.

Можно нажать правой кнопкой мыши не только на файле, но и на папке.




В этом случае пользователь может "Просмотреть" снимок с резервной копией всей папки и файлов в ней и при необходимости восстановить нужные файлы из нее (включая ранее удаленные файлы). Снимок папки открывается в обычном окне проводника Windows. В папке будут данные в состоянии, в котором они были на момент создания снимка.


В результате пользователь может без привлечения администратора сам найти данные, которые ранее были случайно удалены или испорчены и восстановить их. Механизм теневого копирования данных это очень удобный и экономичный с точки зрения использования системных ресурсов способ резервного копирования данных.

Web-интерфейс для работы с резервными копиями

Начиная с версии 1.1.05 в систему может встраиваться Web-интерфейс для просмотра резервных копий, созданных подсистемой теневого копирования данных. Этот вариант просмотра является альтернативным по отношению к тому, что описано в прошлой теме. Web-интерфейс показывает информацию о резервных копиях не только в Windows, но и в любой другой операционной системе и в некоторых случаях информация, которую он показывает является более точной чем то, что показывает Windows.

Web-интерфейс доступен по адресу, указанному в поле "Адрес интерфейса для просмотра резервных копий" в окне свойств теневого копирования.

При открытии этого адреса, появляется только поле ввода, в которое нужно ввести либо путь к папке, либо путь к файлу, резервные копии которого нужно посмотреть.



Можно указывать либо абсолютный, либо относительный путь к файлу. Абсолютный должен выглядеть как  /data/share/<имя-общей-папки>. Если указывается относительный путь, то автоматически подставляется /data/share/files, так как files является общей папкой по умолчанию. Вы можете указывать также путь в стиле Windows, а также букву сетевого диска, например Z:\Документы\договор1.doc преобразуется в /data/share/files/Документы/договор1.doc и покажет информацию о резервных копиях именно этого файла.

Интеграция Web-интерфейса с операционной системой

Не всегда удобно вводить полный путь к файлу или папке, чтобы увидеть его резервные копии. Для обеспечения большего удобства, в комплекте с Web-интерфейсом поставляются скрипты, которые добавляют в контекстное меню стандартного файлового менеджера пункт "Посмотреть резервные копии". Можно нажать правой кнопкой мыши по файлу или каталогу, выполнить эту команду и система автоматически откроет браузер с web-интерфейсом просмотра резервных копий, который будет отображать все резервные копии данного файла или каталога.

Эти скрипты находятся на сервере в общей папке utils, в подкаталоге shadowcopy. В этом каталоге есть подкаталоги "Linux", "MacOS" и "Windows", в каждом из которых находится скрипт для стандартного файлового менеджера соответствующей операционной системы. Необходимо инсталлировать этот скрипт в систему.

Linux

Дистрибутивов Linux очень много и в мире Linux не существует такого понятия как "стандартный файловый менеджер". Однако большинство дистрибутивов по умолчанию используют одну из двух графических оболочек: Gnome и KDE. В графической среде Gnome используется файловый менеджер Nautilus, а в среде KDE по умолчанию используется файловый менеджер Dolphin. Скрипты для этих двух файловых менеджеров можно найти в подкаталогах Gnome и KDE каталога utils/shadowcopy/Linux/ . Мы знаем что существует множество других, менее распространенных файловых менеджеров для GNU/Linux. Если вы заинтересованы в наличии команды "Посмотреть резервные копии" для одного из них, напишите нам об этом по адресу info@lvacompany.ru. Возможно мы сделаем скрипт для него, если это возможно.

Gnome

Стандартным файловым менеджером среды Gnome является Nautilus. Скрипт, добавляющий команду "Посмотреть резервные копии" в контекстное меню этого менеджера находится в каталоге utils/shadowcopy/Linux/Gnome и называется просто "Просмотреть резервные копии". Чтобы установить этот скрипт в системе, нужно скопировать его в домашнюю папку пользователя, в подпапку .gnome2/nautilus-scripts и перезапустить Nautilus. Команда "Посмотреть резервные копии" появится в контекстном меню, в подменю "Сценарии" при щелчке правой кнопкой по файлу или папке.



KDE

Стандартным файловым менеджером среды KDE является Dolphin. Скрипт, добавляющий команду "Посмотреть резервные копии" в контекстное меню этого менеджера находится в каталоге utils/shadowcopy/Linux/KDE и называется openurl.desktop. Необходимо скопировать этот файл в каталог /usr/share/kde4/services/ServiceMenus. После этого команда "Посмотреть резервные копии" появится в контекстном меню Dolphin, при нажатии правой кнопкой мыши на файле или папке. Она появится либо в корне контекстного меню, либо в подменю "Действия", в зависимости от наличия других подобных команд.

MacOS

В MacOS стандартным файловым менеджером является Finder. Скрипт, добавляющий команду "Посмотреть резервные копии" в контекстное меню данного файлового менеджера выполнен в виде службы, которую необходимо зарегистрировать в системе с помощью Automator. Служба представляет из себя каталог "Посмотреть резервные копии.workflow" и находится в каталоге utils/shadowcopy/MacOS. Если скопировать его в систему MacOS, он в ней  будет отображаться не как каталог, а как файл службы Automator и иметь соответствующий значок. Чтобы зарегистрировать эту службу в системе, следуйте инструкции:

1. Запустите Automator из списка стандартных приложений MacOS

2. Выберите "Служба" в качестве шаблона процесса



3. Выполните команду Файл-->Открыть из меню и откройте службу "Посмотреть резервные копии", (каталог которой вы ранее скопировали). Содержимое службы появится на экране



В тексте скрипта можно откорректировать IP-адрес и порт, на котором находится Web-интерфейс для просмотра резервных копий.

4. Выполните команду Файл --> Сохранить как, далее введите имя, под которым эта служба будет зарегистрирована (или оставьте по умолчанию "Посмотреть резервные копии")

5. Закройте Automator. Команда "Посмотреть резервные копии" должна появиться в контекстном меню Finder при щелчке правой кнопкой мыши по файлу или папке.



Если после выполнения всех операций команда не появилась в контекстном меню, попробуйте перезагрузить систему.

Windows

Стандартным файловым менеджером Windows является Windows Explorer, который чаще называют просто "Проводник". В подкаталоге utils/shadowcopyWindows/openurl находится скрипт, добавляющий команду "Посмотреть резервные копии" в проводник Windows. Нужно скопировать весь каталог openurl на компьютеры пользоватлей в корень диска C:, войти в него и запустить файл "openurl.reg". Этот файл внесет необходимые изменения в реестр, после чего пункт "Показать резервные копии" появится в контекстном меню Проводника Windows при щелчке правой кнопкой мыши по папке или файлу.
